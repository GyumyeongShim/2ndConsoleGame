# 프로젝트 리팩토링 제안서 (Refactoring Proposals)

코드베이스 분석 결과, 시스템의 안정성, 확장성 및 유지보수성 향상을 위해 개선이 필요한 리팩토링 항목들을 다음과 같이 제안합니다.

---

## 1. 메모리 관리 및 소유권 모델 현대화 (Memory Management)

*   **현상:** `Actor*`, `Component*` 등 생 포인터(Raw Pointer)를 사용한 수동 메모리 관리와 `new/delete`가 산재해 있어 메모리 누수 위험이 있음.
*   **개선안:** 
    *   `std::unique_ptr` 및 `std::shared_ptr` 적극 도입.
    *   `Level` 클래스가 액터의 소유권(`unique_ptr`)을 명확히 가지고, 외부 시스템은 관찰자(Raw Pointer 또는 `weak_ptr`)로서만 참조하도록 구조 변경.
*   **기대 효과:** 메모리 누수 및 댕글링 포인터(Dangling Pointer) 문제 근본적 해결.

## 2. 거대 클래스(God Object) 분리 및 책임 분산 (Decomposition)

*   **현상:** `Actor` 및 `BattleLevel` 클래스가 너무 많은 역할(데이터, 로직, UI 제어)을 수행함.
*   **개선안:**
    *   **Actor 경량화:** 장비 로직은 `EquipmentComponent`, 이동 로직은 `MovementComponent`로 완전히 위임.
    *   **State 패턴 도입:** `BattleLevel`의 거대한 `switch-case` 및 `Phase_...` 메서드들을 독립된 `State` 객체(예: `CommandState`, `AnimationState`)로 분리.
*   **기대 효과:** 클래스 단위 테스트 용이성 확보 및 코드 가독성 향상.

## 3. UI와 게임 로직의 디커플링 (Decoupling)

*   **현상:** `BattleLevel`이 구체적인 UI 클래스들에 강하게 결합되어 있어 UI 수정 시 레벨 로직까지 영향을 받음.
*   **개선안:**
    *   **이벤트 시스템(Observer Pattern) 도입:** 전투 사건(데미지 발생, 턴 변경 등) 발생 시 이벤트를 발행하고, UI는 이를 구독하여 자율적으로 갱신.
    *   **UIManager 구축:** 모든 UI 요소의 생성과 파괴를 담당하는 중앙 관리자 도입.
*   **기대 효과:** UI 교체 및 연출 수정 시 게임 로직의 안정성 보장.

## 4. 컴포넌트 시스템 최적화 (Performance)

*   **현상:** `GetComponent<T>()` 호출 시마다 `dynamic_cast`를 사용하며, 이는 빈번한 호출 시 성능 저하를 유발함.
*   **개선안:**
    *   **Type ID 기반 검색:** 각 컴포넌트 클래스에 고유한 정수 ID 또는 비트마스크를 부여하여 캐스팅 없이 즉시 검색 가능하게 개선.
*   **기대 효과:** 프레임당 다수의 액터가 컴포넌트를 참조하는 환경에서 성능 최적화.

## 5. 데이터 중심 설계(Data-Driven) 확장 (Flexibility)

*   **현상:** 몬스터의 AI 패턴이나 특정 연출 효과가 코드 내에 하드코딩되어 있어 콘텐츠 추가 시마다 리컴파일이 필요함.
*   **개선안:**
    *   **AI 패턴 데이터화:** 몬스터의 행동 트리나 조건부 스킬 사용 로직을 JSON 설정 파일로 추출.
    *   **Constants 중앙화:** `SortingOrder`, `MaxSlots` 등 매직 넘버들을 `Config/Setting.txt`로 통합 관리.
*   **기대 효과:** 프로그래머의 개입 없이 기획 데이터 수정만으로 게임 밸런싱 및 콘텐츠 확장 가능.

## 6. 빌드 시간 및 의존성 최적화 (Compilation)

*   **현상:** 헤더 파일 간의 상호 참조(#include)가 많아 소스 파일 하나 수정 시 대규모 재컴파일이 발생함.
*   **개선안:**
    *   **전방 선언(Forward Declaration) 활용:** 헤더 파일에서는 클래스 이름만 선언하고, 실제 참조는 `.cpp` 파일에서 수행.
    *   **PIMPL 패턴 고려:** 노출이 잦은 핵심 클래스에 구현부 분리 패턴 적용.
*   **기대 효과:** 프로젝트 규모 증가 시에도 쾌적한 빌드 속도 유지.

---

## ✅ 권장 우선순위
1.  **메모리 관리 개선:** 시스템 안정성의 근간이므로 최우선 적용 권장.
2.  **거대 클래스 분리 (State 패턴):** 현재 가장 복잡한 전투 로직의 유지보수성을 위해 시급함.
3.  **이벤트 기반 UI 구조 변경:** 연출 퀄리티 향상을 위한 필수 단계.
